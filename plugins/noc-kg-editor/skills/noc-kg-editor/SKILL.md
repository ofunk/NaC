# NoC KG Editor

## Operating Boundary

Runtime mode: `local-kg-editor-companion`.

This plugin is for notary-facing work with usecase-local knowledge graphs. It
opens the safe editor view generated by `notary_kg.editor` and presents it as a
form/checklist surface. Do not ask Fachpersonal to edit raw JSON.

## Allowed Work

- Open one usecase KG by slug.
- Render the editor view as text, JSON or Markdown.
- Explain open information, documents, decisions, gates and evidence.
- Prepare a structured patch proposal for review.
- Validate a patch before a pull request.
- Create a pull request only after explicit human confirmation.

## Prohibited Work

- Store real mandate, client, family, health, estate, property or secret values
  in Git.
- Edit `value` fields in KG files.
- Apply blind overwrites to `knowledge-graph.graph.json`.
- Treat LLM output as fachliche Wahrheit without review and merge.
- Send local workstation, XNP, card-reader, morris, portal or credential values
  to a public action endpoint.

## Workflow

1. Identify the usecase slug, for example `adoption-familienrechtliche-erklaerungen`.
2. Open the editor view:

   ```powershell
   python plugins\noc-kg-editor\scripts\open_editor.py <slug>
   ```

3. For app or bridge integration, request JSON:

   ```powershell
   python plugins\noc-kg-editor\scripts\open_editor.py <slug> --format json
   ```

4. For a Markdown session sheet, write to `out/kg-editor/`:

   ```powershell
   python plugins\noc-kg-editor\scripts\open_editor.py <slug> --format markdown --output out\kg-editor\<slug>.md
   ```

5. If changes are requested, propose a schema-valid patch, show the diff, run
   validation, ask for confirmation, then create a pull request.

## Output Shape

Return concise sections named `Usecase`, `Editor Tabs`, `Patch Boundary`,
`Next Operator Step` and `Validation`. If a usecase slug is unknown, list the
expected slug pattern and do not guess a replacement.
