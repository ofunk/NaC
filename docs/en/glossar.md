# Glossary For Non-Technical Readers

This glossary explains NoC terms in the way they matter for notary-office
operations.

| Term | Meaning For The Office |
| --- | --- |
| Git | The versioned protocol. It shows who changed which rule, template or decision and when. |
| Repository | The shared working folder with rules, usecases, workflows, tests and evidence. |
| Private fork | The office's own non-public copy for a real office or pilot. Local roles and operating rules live there. |
| Pull request | A reviewed change proposal. It replaces unreviewed direct edits to important rules. |
| Review | Subject-matter or technical checking before a change becomes binding. |
| Merge into `main` | The change enters the binding main state. |
| Knowledge Graph / KG | The structured list of open information, documents, decisions, gates and evidence for a case type. |
| Usecase | A concrete notarial case type, for example real-estate purchase contract or GmbH formation. |
| Workflow | A repeatable flow of input, check, approval and evidence. |
| Plugin | A companion package for a technical or subject-matter capability, for example card-reader readiness or register path. |
| Quality gate | An automated check run. It shows whether tests, privacy rules, language rules and KG rules are satisfied. |
| SBOM / AI-SBOM | A supply-chain list for software, models, external services and local components. |
| Evidence | Evidence or an evidence reference. The public repo contains metadata only, not real mandate documents. |
| Readiness | A readiness check. It says whether a local workstation or plugin path is technically prepared. |
| Dry run | A plan preview without productive write action and without real mandate data. |

## Rule Of Thumb

NoC makes rules, open questions and approvals visible. It is not the place for
real file contents.
